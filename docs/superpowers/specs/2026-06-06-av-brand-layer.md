# Alta Vibración brand layer over @saas/ui

**Date:** 2026-06-06
**Status:** Decided
**Spike:** [oscar-ospina/saas-planner#18](https://github.com/oscar-ospina/saas-planner/issues/18)
**Parent epic:** [oscar-ospina/saas-planner#16](https://github.com/oscar-ospina/saas-planner/issues/16)

## Decision

For the Alta Vibración consumer app, use a **thin brand layer**: inherit `@saas/ui`'s shipped theme as-is and add only **(a)** self-hosted fonts (Archivo + Open Sans), **(b)** the logo SVG lockups, and **(c)** brand-specific component *compositions* that live in the app, not the DS (Hero, WhatsApp FAB, section eyebrow, consultation cards, navbar/footer).

**Do not** refactor `@saas/ui` to neutral tokens now. Defer an explicit "neutral base + named brand theme" split until a second brand actually exists (YAGNI).

## Context

The load-bearing product principle (see `product-status` memory / [README](../../../README.md#the-product)) is: **`@saas/ui` is brand-agnostic; Alta Vibración is the reference brand / first consumer, not baked in.** This spike was raised because an early reading suggested `@saas/ui@0.2.0` "bakes the AV palette in," contradicting that principle. That reading came from a **stale pre-0.2.0 copy** inside the design bundle; the claim was checked against the live source and corrected.

### Verified architecture of `@saas/ui@0.2.0` (read 2026-06-06 from `saas-packages/ui/src/`)

Two clean layers:

- **`tokens.css`** (generated by the token pipeline) — the raw palette ramps with literal AV hexes: `--color-orange-*` (orange-400 = `#f37d3e`), `--color-violet-*` (violet-500 = `#7f5af8`), `--color-neutral-*`, `--color-text-*`, surfaces, semantic red.
- **`semantic.css`** — maps shadcn **roles** onto those tokens: `--color-primary` → `orange-300`, `--color-primary-foreground` → `orange-900` (the AA-safe pair), `--color-ring` → `orange-500`, `--color-background`/`--color-foreground`/`--color-destructive`, plus derived neutral roles (border/input/muted/accent).

**Components consume the semantic roles (`bg-primary`, `ring-ring`, …), not raw hexes.** So the DS *is* brand-agnostic **in structure** — the principle holds. The only nuance vs the literal "not baked in" wording: **AV is shipped as the single default theme** (there is no neutral default + named-brand theme to swap), so a *second* brand would have to override the `@theme` vars to reskin.

### The override pattern already exists

The stack ADR ([#6](2026-05-27-ds-stack-decision.md), §"Theme composition and overrides") already established that consumers override individual tokens by redeclaring the same CSS variable in their own `@theme` block (later declaration wins in the cascade), with per-app brand variation via a `--brand-*` namespace. So the AV app has a documented escape hatch if it ever needs to diverge from the DS defaults.

## Options

- **A — Thin layer (CHOSEN).** App inherits the AV defaults from `@saas/ui` + adds fonts, logo, and brand component compositions. May redeclare a handful of `@theme` vars via the documented override pattern if it wants to diverge.
- **B — Neutral refactor of `@saas/ui`.** Rename `tokens.css`/`semantic.css` to neutral semantic names and ship an AV theme (`themes/alta-vibracion.css`) the app imports. True multi-brand drop-in.

## Rationale for A

- AV is the **reference brand and the only consumer** today — the shipped defaults are exactly what the app wants; re-skinning is not a current need.
- **B has no driver** (no second brand) and would churn the just-shipped `0.2.0`: re-baselining the Playwright VR snapshots and a semver bump, for zero present value → YAGNI.
- Because components already use **semantic roles**, B stays a clean, *additive* change whenever it's actually needed (introduce neutral defaults + AV as a named theme) **without touching component source**. Deferring costs nothing structurally.

## Consequences

- Scopes story [#20](https://github.com/oscar-ospina/saas-planner/issues/20) (Brand theme layer) to: self-host Archivo + Open Sans and wire them to `--font-display` / `--font-sans`; ship the `logo-horizontal.svg` / `logo-mark.svg` lockups; build brand component compositions in the app. No `@saas/ui` change required.
- Light-only (no dark palette — deferred theming epic), consistent with `@saas/ui@0.2.0`.

## When to revisit B

Trigger: **a second brand/consumer appears**, or a decision to publish `@saas/ui` as a neutral public library. At that point, file a story in `saas-packages` to refactor `tokens.css` to neutral semantic names and add `themes/alta-vibracion.css`. Until then, AV-as-default is the right shape.
